# BoxMail: A Box-Integrated Email Client — Comprehensive Plan & Roadmap

The repository is currently empty (only `LICENSE` and `README.md`), so this plan is greenfield. The vision: an email client where every attachment, every shared link, every contact, and every workflow is natively wired into Box Drive, Box Sign, Box Hubs, Box Relay, and Box AI — making Box the "system of record" for everything that flows through email.

---

## Guiding Principles

1. **Box-first, not Box-bolted-on.** Attachments default to Box links; "downloads" land in Box; collaborators are Box-aware.
2. **IMAP/SMTP/JMAP-agnostic mail layer.** The mail engine must be provider-neutral (Gmail, Microsoft 365, generic IMAP) so Box integration is the differentiator, not the mail backend.
3. **Zero-trust, least-privilege OAuth.** Both the mail provider and Box hold their own scoped tokens; the client never proxies credentials.
4. **Local-first, cloud-synced.** A local cache (SQLite + object store) for offline; Box and mail server are the sources of truth.
5. **Extensible.** Plugin model so Box features (Sign, Hubs, AI, Relay, Skills) can be added without forking the core.

## Epic 1 — Foundation & Architecture

**Goal:** Establish the project skeleton, tech stack, and cross-cutting concerns before any feature work.

Overarching features:
- Choose deployment surface: desktop (Electron/Tauri), web (React/Next), and/or mobile (React Native). Recommend Tauri + shared TypeScript core for performance and small footprint.
- Monorepo layout (e.g., `packages/core`, `packages/mail`, `packages/box`, `apps/desktop`, `apps/web`).
- Shared domain model (Message, Thread, Contact, Attachment, BoxItem, Collaboration).
- Local storage layer: SQLite (FTS5 for search), encrypted at rest, with a blob cache for message bodies and thumbnails.
- Background sync engine with a job queue (retry, backoff, conflict resolution).
- Logging, telemetry (opt-in), crash reporting, feature flags.
- CI/CD, code signing, auto-update channel.
- Security baseline: secret storage in OS keychain, CSP, sandboxed renderers, dependency scanning.

## Epic 2 — Mail Protocol & Account Layer

**Goal:** A robust, provider-agnostic mail engine.

Overarching features:
- Account onboarding for Gmail (OAuth/Gmail API), Microsoft 365 (Graph API), and generic IMAP/SMTP (with OAuth2/XOAUTH2 where possible).
- JMAP support for forward-looking providers (Fastmail).
- Folder/label model that normalizes Gmail labels vs. IMAP folders.
- Incremental sync: IMAP IDLE, Gmail history API, Graph delta queries.
- Send pipeline: SMTP submission, Gmail/Graph send endpoints, draft sync, scheduled send, undo send.
- MIME parsing/composition (including S/MIME and PGP hooks for later).
- Threading/conversation grouping (References/In-Reply-To + subject heuristic).
- Rich-text composer (HTML + plaintext fallback), signatures, templates.

---

## Epic 3 — Box Authentication & SDK Integration

**Goal:** Make Box a first-class identity and storage layer.

Overarching features:
- Box OAuth 2.0 with PKCE for desktop/mobile; JWT or CCG for any server-side components.
- Token refresh, revocation, and multi-account Box support (work + personal).
- Box Node/JS SDK wrapper in `packages/box` exposing a typed, promise-based API surface.
- Webhook receiver (or polling fallback) for Box events (file updated, comment added, collaboration accepted, Sign completed).
- Box App Center / Box Skills registration considerations for the enterprise install path.
- Admin console settings (for Enterprise) — choose default root folder ("BoxMail/"), retention policy, shared-link defaults, classification labels.

---

## Epic 4 — Attachments via Box (the headline feature)

**Goal:** Every attachment is a Box object, inbound and outbound.

Overarching features:
- **Outbound:** Drag-and-drop or "Attach from Box" picker; large files auto-uploaded to a per-thread Box folder; insertion of Box shared links with configurable access (company, collaborators, password, expiration, download permission).
- **Inline previews:** Use Box's preview/thumbnail/embed URLs so recipients see rich previews even outside Box.
- **Inbound:** When an email arrives with attachments, auto-save to a structured folder (e.g., `BoxMail/<year>/<sender domain>/<thread subject>/`), with original message metadata. Replace MIME parts with Box links in the local view (configurable).
- **Deduplication:** SHA-1/content hash check against Box to avoid re-uploading.
- **Large-file uploads:** Use Box Chunked Upload API for files >50 MB with resume support.
- **Versioning:** Subsequent revisions of the same attachment become new Box file versions, not duplicates.
- **Quarantine/scan:** Optional malware scanning hook before saving to Box.

## Epic 5 — Deep Box Drive Integration

**Goal:** Make the user's Box folder structure visible and actionable inside the mail client.

Overarching features:
- Embedded Box file browser pane (Box UI Elements `ContentExplorer`, `ContentPicker`, `ContentPreview`).
- "Send as Box link" command from any Box item (right-click → email).
- Bi-directional context: open an email and see related Box folders/files; open a Box file and see related emails.
- Activity timeline merging email events + Box events (file edited, comment added, Sign completed) for a given collaborator or project.
- Smart folders: auto-create a Box folder per email thread, per sender, or per project tag.

## Epic 6 — Collaboration & Contact Intelligence

**Goal:** Treat people as a unified Box + email identity.

Overarching features:
- Unified contact card: email address(es), Box user ID, recent shared files, active collaborations, last interaction.
- When composing to a recipient, surface "files you've recently shared with them" and "files you both have access to."
- One-click "invite as Box collaborator" from any email recipient, with role selection.
- Group/distribution list awareness mapped to Box groups.
- Enterprise directory sync (SCIM/Graph) to resolve internal users.

## Epic 7 — Box Sign Integration

**Goal:** Native e-signature flow from the inbox.

Overarching features:
- "Request signature" action on any inbound attachment or composed message.
- Sign template picker, signer assignment from email recipients, signing-order configuration.
- Status tracking in the message thread (sent, viewed, signed, declined, expired).
- Completed signed PDFs auto-saved to the thread's Box folder with audit log.
- Reminders and re-send via Sign API.

## Epic 8 — Box Hubs, Relay & Workflow Automation

**Goal:** Turn email into structured workflows.

Overarching features:
- **Box Hubs:** Curate a Hub of all email-derived assets for a customer/deal/case; one click to publish a thread's Box folder into a Hub.
- **Box Relay:** Trigger pre-built workflows from email actions (e.g., new RFP attachment → start review workflow; signed NDA → move to "Active Customers" Hub).
- **Custom rules engine:** "If sender domain = X and subject contains Y, save attachments to folder Z, apply classification, start Relay workflow W."
- **Metadata templates:** Apply Box metadata (deal stage, project ID, retention class) to email-derived files automatically.

## Epic 9 — Box AI & Intelligent Inbox

**Goal:** Leverage Box AI plus modern LLMs for inbox productivity, grounded in user's Box content.

Overarching features:
- Thread summarization (Box AI for context that includes related Box files; mail-only summaries via a chosen LLM).
- "Ask about this email" Q&A grounded in linked Box files (RAG over Box content via Box AI API).
- Smart compose / smart reply with context from the user's Box documents (e.g., pull pricing from a Box spreadsheet).
- Auto-classification: project/customer/intent tagging, with the tag stored as Box metadata.
- Triage suggestions: "Save attachment to project folder," "Start Relay workflow," "Schedule follow-up."
- Translation, action-item extraction, meeting prep digest.

## Epic 10 — Search & Discovery

**Goal:** One search bar across mail + Box.

Overarching features:
- Federated search: local FTS5 over mail + Box Search API for files/comments/metadata, merged and ranked.
- Filters: from, to, has-attachment, in-box-folder, has-sign-request, file-type, date, classification.
- Saved searches and smart folders.
- Natural-language search powered by Box AI ("emails from Acme last quarter with signed contracts").

## Epic 11 — Security, Compliance & Governance

**Goal:** Be enterprise-deployable on day one.

Overarching features:
- Box classification labels propagated to outbound emails (warnings, restricted recipients).
- DLP hooks: block sending if attachment classification is too high for external recipients.
- Box retention policies honored for email-derived files; legal hold awareness.
- Box Shield alerts surfaced in-client (suspicious download of file shared via email).
- End-to-end encryption optional (S/MIME, PGP); Box KeySafe compatibility for enterprises using their own keys.
- Audit log of every Box action taken from the client (uploads, link creations, permission changes).
- GDPR/CCPA tooling: data export, account deletion, region-pinned storage (use Box Zones).
- SOC 2 / ISO 27001 readiness from the start; threat model and pen-test plan.

## Epic 12 — Offline, Sync, Performance

**Goal:** Feels instant; works on a plane.

Overarching features:
- Offline composer with queued sends; offline Box link generation deferred until reconnect.
- Pre-fetch attachments for starred/recent threads to local Box-cache.
- Conflict resolution when the same Box file is modified offline and online (use Box's `If-Match`/etag, prompt user).
- Bandwidth-aware sync; pause on metered connections.
- Snappy UI: virtualized lists, optimistic updates, background indexer.

## Major Challenges & Risks

1. **Mail provider heterogeneity.** Gmail labels, Exchange categories, and IMAP folders all behave differently; building a clean abstraction that doesn't leak provider quirks is hard. Mitigation: a provider-adapter pattern with rigorous conformance tests.

2. **Attachment semantics rewriting.** Replacing MIME attachments with Box links changes what recipients see — especially non-Box users, mailing lists, and archival systems. Need fallback strategies (e.g., attach small files normally, link only when >N MB or when policy says so), and clear UX about who can access the link.

3. **Shared-link permission model mismatch.** Email is "send and forget"; Box links are revocable, expirable, and access-controlled. Reconciling these mental models — and preventing accidental over-sharing — is a UX and security challenge.

4. **OAuth token sprawl.** Two (or more) refreshable token sets per account, each with different expiry/refresh semantics. Need a unified token vault and proactive refresh.

5. **Sync correctness.** Bi-directional sync between local cache, mail server, and Box folder is a classic distributed-systems problem. Etags, conflict prompts, and idempotent operations are essential.

6. **Large-scale historical import.** A user with 20 GB of attachments will hit Box upload quotas, rate limits, and bandwidth caps. Must be throttled, resumable, and optional.

7. **Box API rate limits & quotas.** Bursty workloads (mass send, bulk import, Hubs publishing) can trip limits. Need a global rate-limit governor and exponential backoff per endpoint class.

8. **Search relevance across two indexes.** Mail FTS and Box Search rank differently; building a merged ranking that feels "right" requires tuning and user feedback.

9. **Security & compliance surface area.** S/MIME, PGP, DLP, classification, retention, legal hold, KeySafe — each is a project on its own. Sequence carefully; don't ship half-built compliance.

10. **AI grounding & hallucination.** Smart compose using Box documents must cite sources and avoid leaking restricted content into prompts. Need per-file ACL checks before context injection.

11. **Privacy of email content in Box.** Storing email bodies/attachments in Box means email content now inherits Box governance — useful, but users must consent and admins must understand the implications.

12. **Cross-platform parity.** Desktop, web, mobile each have different storage, background-execution, and OS-integration models (e.g., default mail handler, mailto: protocol, share sheets).

13. **Deliverability.** Sending via the user's own provider (Gmail/SMTP) preserves deliverability; sending via a relay would create SPF/DKIM/DMARC headaches. Commit to user-provider sending.

14. **Box product evolution.** Box AI, Hubs, and Relay APIs are evolving; build behind feature flags and abstract behind interfaces so API changes don't ripple.

15. **Enterprise procurement.** Selling into IT requires SSO (SAML/OIDC), SCIM, audit log streaming (to Splunk/Sentinel), and DPA paperwork — non-trivial GTM cost.

16. **Performance with very large mailboxes.** 100k+ message accounts stress threading, search, and UI virtualization. Profile early; don't paint yourself into an architectural corner.

17. **Migration from existing clients.** Users won't switch unless import (mbox, PST, Gmail Takeout) and contact/calendar migration are seamless.

## Follow-up Thoughts on using Coding Agents to Implement

This plan describes a 12–18+ month engineering effort spanning 15 epics, multiple native platforms, two independent OAuth ecosystems, real-time sync engines, a plugin SDK, enterprise compliance features, and integration with at least 8 distinct Box product APIs. 

Concretely:
- Scope is orders of magnitude too large for a single agent session. A one-shot run would mean hundreds of thousands of lines of production code, configuration, tests, and infrastructure. Claude's context window is finite; it would lose coherence long before finishing even Phase 1.
- Many steps require you, not an agent. Registering a Box developer app, configuring OAuth redirect URIs, obtaining Gmail/Microsoft API credentials, provisioning infrastructure, setting up code-signing certificates, choosing a billing provider — these require human decisions and accounts that an agent cannot act on.
- The plan intentionally front-loads discovery. Phase 0 is UX research and technical spikes. Those spikes produce information that changes the architecture; you can't skip them and write the final code speculatively.
- Correctness and security require iteration. An email client that touches OAuth tokens, shared-link permissions, DLP rules, and user mailboxes cannot be "shipped" by a single automated pass. It needs code review, security testing, and real-world usage feedback.

What Claude can usefully do, session by session:
- Scaffold the monorepo and Tauri/Electron shell (Epic 1).
- Wire up Box OAuth + SDK in packages/box (Epic 3).
- Implement the Gmail adapter (Epic 2, Gmail slice).
- Build the outbound Box attachment flow (Epic 4 core).
- Write tests for any of the above.

The right approach is to use Claude as a fast, capable pair programmer for each bounded task — not as a one-shot factory for the whole application. Treat each session as one sprint or one module, keep PRs small, and review the output before moving on.
